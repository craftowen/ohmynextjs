# Team Lead with Auto-Rotation

사용자가 요청한 작업을 **리서치 → 복잡도 판단 → 플랜(승인) → 실행 → 검증 → 마무리** 사이클로 완수한다.
팀원은 `max_turns` 도달 시 자동 교체하고, 리더 기억은 handoff-log.md에 보존한다.

인자로 전달된 내용이 작업 설명이다. 즉시 Phase 0부터 시작하라.

---

## Phase 0: 도구 로드 (필수 — 반드시 Phase 1 전에 실행)

**이 스킬은 deferred tool을 사용한다. Phase 1에 들어가기 전에 반드시 아래 도구들을 ToolSearch로 로드하라.**
로드하지 않으면 TeamCreate 등의 도구를 호출할 수 없어 팀 생성이 실패한다.

**배치 로드** — 2회 병렬 호출로 모든 도구를 로드한다:

1. `ToolSearch("select:TeamCreate,TeamDelete,TaskCreate,TaskList,TaskGet")`
2. `ToolSearch("select:TaskUpdate,TaskStop,TaskOutput,SendMessage,AskUserQuestion")`

**모든 도구 로드 완료 후** Phase 1로 진행한다.

---

## 전체 흐름

```
Phase 1: 셋업       팀 생성, git checkpoint, 세션 재개 감지
   ↓
Phase 2: 리서치     Explore 팀원 조사 → 결과 취합 → ★ 복잡도 판단 (S/M/L/XL)
   ↓
Phase 3: 플랜       복잡도 기반 깊이로 계획 수립, 파일 소유권 매핑 → 사용자 승인
   ↓
Phase 4: 실행       복잡도 기반 팀 규모, 파일 충돌 방지, 중간 보고
   ↓
Phase 5: 검증       빌드/테스트 자동검증, 코드 리뷰, 에러 복구 루프
   ↓
Phase 6: 마무리     최종 보고, 커밋 제안, /deploy-dev 연동
```

**팀원 교체는 어느 Phase에서든 HANDOFF 수신 또는 max_turns 도달 시 즉시 실행.**

## Phase 게이트 요약

| Phase | Exit Criteria | 실패 시 |
|-------|--------------|---------|
| 1 셋업 | 팀 생성 + handoff-log 존재 | 재시도 1회 → 에스컬레이션 |
| 2 리서치 | 복잡도 결정 + handoff-log 기록 | 추가 리서치 팀원 spawn |
| 3 플랜 | 계획 수립 완료 | 모호할 때만 사용자 질문 |
| 4 실행 | 모든 태스크 completed | HANDOFF 교체 / 에러 복구 |
| 5 검증 | 검증 게이트 4항목 ALL PASS | 에러 복구 루프 (max 3회) |
| 6 마무리 | 보고 완료 + 팀 삭제 | - |

## Phase 1: 셋업

### 1.1 세션 재개 감지
`~/.claude/teams/` 에서 기존 팀 존재 여부 확인:
- **기존 팀 있음** → handoff-log.md의 `repo_root`, `branch` 메타데이터가 현재 작업 환경과 일치하는지 검증
  - **repo + branch 모두 일치** → TaskList 읽어 상태 파악 → 사용자에게 재개 여부 확인
  - **repo 일치, branch 불일치** → 사용자에게 확인: "같은 repo에서 {branch}로 작업하던 팀이 있습니다. 재개/정리 중 선택해주세요."
  - **repo 불일치** → 무관한 팀으로 간주, 신규 팀 생성 (기존 팀은 건드리지 않음)
- **기존 팀 없음** → 신규 팀 생성
- **좀비 팀 감지**: handoff-log.md에 `last_active` 타임스탬프가 24시간 이상 지난 팀 → 사용자에게 정리 제안

### 1.2 팀 생성
TeamCreate로 팀을 생성한다. 팀 이름은 작업 내용에서 유추하여 간결하게 짓는다.

### 1.3 Git Checkpoint
작업 시작 전 복원 지점 기록:
- Working tree dirty → `git stash push -m "team-checkpoint-{timestamp}"`
- Clean → 현재 HEAD hash를 handoff-log에 기록

### 1.4 Handoff Log 초기화
`~/.claude/teams/{team-name}/handoff-log.md` 파일을 생성:

```markdown
# Handoff Log - {team-name}
> 리더의 외부 기억 저장소.
> 생성: {날짜시간}

## 작업 개요
{사용자 요청 요약}

## 세션 메타데이터
- Repo: {git repo root 절대경로}
- Branch: {branch}
- last_active: {ISO 8601 타임스탬프}

## Git Checkpoint
- Commit: {hash}
- Stash: {stash ref 또는 "N/A"}

## 복잡도
(Phase 2 완료 후 기입)

## 리더 Phase 추적
- 현재 Phase: 1
- Phase 전환 이력: Phase 1 시작 ({시간})

## 파일 변경 추적
| 파일 | 담당 팀원 | 상태 |
|------|----------|------|
(Phase 3 이후 업데이트)
```

### 1.5 리서치 팀원 Spawn
작업 범위에 따라 1~N명의 리서치 팀원을 spawn.
- 각 팀원에게 조사할 영역을 명확히 분배
- **모든 팀원 프롬프트에 max_turns 교체 프로토콜과 프로젝트 컨텍스트를 포함**

---

## Phase 2: 리서치

### 2.1 리서치 팀원 Agent 설정

| 역할 | subagent_type | model | max_turns | 용도 |
|------|--------------|-------|-----------|------|
| 코드 조사 | `Explore` | `haiku` | 15 | 파일 패턴 탐색, 키워드 검색 |

### 2.2 리서치 태스크 생성 & 할당
조사 영역별로 TaskCreate → 팀원에게 할당.
예: 코드베이스 구조, 기존 패턴, 관련 파일, 의존성 등.
팀원은 조사 결과를 **아래 구조화된 보고 포맷**으로 리더에게 메시지 보고.

**리서치 보고 포맷 (팀원 프롬프트에 포함):**
```
## 리서치 결과: {조사 영역}

### 영향 파일 목록
| 파일 경로 | 수정 필요 여부 | 이유 |
|----------|--------------|------|
| `path/to/file` | Yes/No | {이유} |

### 발견된 패턴
- {기존 코드에서 발견한 패턴/컨벤션}

### 의존성
- {이 영역이 의존하는 다른 모듈/서비스}
- {이 영역에 의존하는 다른 모듈/서비스}

### 리스크/주의사항
- {발견된 잠재 문제}
```

### 2.3 결과 취합
1. 팀원 보고의 **영향 파일 목록과 리스크**만 handoff-log.md에 기록 (전문 복사 금지 — 로그 비대화 방지)
2. 결과를 종합하여 전체 그림 파악
3. 추가 조사 필요 시 팀원에게 후속 지시

### 2.4 복잡도 판단
리서치 결과를 기반으로 복잡도 등급 결정 → handoff-log에 기록:

| 등급 | 기준 | Phase 3 깊이 | Phase 4 팀원 수 | 팀원 max_turns |
|------|------|-------------|---------------|---------------|
| **S** | 파일 3개 이하, 단일 레이어 | 간소 플랜 (승인은 유지) | 1명 | 20 |
| **M** | 파일 4-10개, 단일 스택 | 표준 플랜 | 2명 | 25 |
| **L** | 파일 11-30개, 풀스택 | 상세 플랜 | 3명 | 30 |
| **XL** | 파일 30개+, 아키텍처 변경 | 상세 플랜 + 중간 승인 | 4명 (최대) | 30 |

**복잡도 상향 조정 규칙:** 파일 수 기준 외에 다음 고위험 영역이 포함되면 복잡도를 한 단계 올린다:
- DB 스키마 변경 (마이그레이션 포함)
- 결제/과금 로직 수정
- 인증/권한 로직 수정
- 외부 API 연동 변경 (LLM 체인, 결제 게이트웨이 등)

### 2.5 리서치 팀원 정리
리서치 완료 → 팀원들에게 shutdown_request.

---

## Phase 3: 플랜

### 3.1 실행 계획 수립
리서치 결과 + 복잡도 등급 기반으로 작성:

```markdown
## 실행 계획

### 복잡도: {S/M/L/XL}

### 목표
{달성할 것}

### 리서치 요약
{핵심 발견사항}

### 태스크 목록
1. {태스크1} - {담당}
2. {태스크2} - {담당}

### 파일 소유권 테이블
| 태스크 | 소유 파일 | 담당 agent | 의존관계 |
|--------|----------|-----------|---------|
| T1 | `src/app/api/foo/route.ts` | agent-1 | - |
| T2 | `src/lib/foo.ts` | agent-1 | - |
| T3 | `src/components/Foo.tsx` | agent-2 | T1 완료 후 |

### 태스크 의존관계
- {의존 관계}

### 예상 결과물
- {파일/기능}

### 리스크
- {잠재 문제점}
```

**파일 소유권 규칙:**
- 동일 파일을 수정하는 태스크는 반드시 `addBlockedBy`로 직렬화
- 한 파일은 한 시점에 한 팀원만 수정 가능

### 3.2 바로 실행
계획을 간략히 출력하고 **승인 없이 즉시 Phase 4로 진행한다.**
- 계획이 명확하면 → 바로 실행
- 선택지가 여러 개이고 어느 쪽이 맞는지 모호할 때만 → AskUserQuestion으로 사용자에게 질문

## Phase 4: 실행

### 4.1 실행 팀원 Agent 설정

| 역할 | subagent_type | model | max_turns | 용도 |
|------|--------------|-------|-----------|------|
| 구현 | `general-purpose` | `sonnet` | 복잡도별 (2.4 참조) | 코드 작성, 파일 수정 |

### 4.2 실행 태스크 생성
승인된 플랜 → TaskCreate. 의존관계는 addBlockedBy로 설정.

### 4.3 실행 팀원 Spawn & 할당
복잡도 기반 팀원 수에 맞춰 spawn.
병렬 가능한 태스크는 동시에 진행.

**팀원 프롬프트 필수 포함:**
- 프로젝트 컨텍스트 (아래 템플릿 참조)
- max_turns 교체 프로토콜
- **파일 충돌 방지 규칙: "할당된 파일 외 수정 금지. 다른 파일 수정 필요 시 리더에게 요청."**

### 4.4 실행 관리 루프
모든 실행 태스크 completed까지 반복:

- **진행 보고** → 확인, 피드백
- **HANDOFF** → 팀원 교체 프로세스
- **태스크 완료** → completed 처리, handoff-log 기록, 다음 태스크 할당
- **블로커** → 해결
- **선행 태스크 완료** → 후속 태스크 할당

### 4.5 중간 보고
복잡도별 보고 주기:

| 복잡도 | 보고 주기 |
|--------|----------|
| S | 완료 시 1회 |
| M | 태스크 2개 완료마다 |
| L | 태스크 2개 완료마다 + 팀원 교체 시 |
| XL | 태스크 1개 완료마다 + 팀원 교체 시 |

```
[진행 상황] 4/8 태스크 완료
- ✅ T1: {설명}
- ✅ T2: {설명}
- 🔄 T3: {진행 중}
- ⏳ T4: {대기}
```

### 4.6 팀원 간 통신
- **모든 통신은 리더 경유** — 팀원끼리 직접 메시지 금지
- 팀원이 다른 팀원의 작업 결과가 필요하면 리더에게 요청
- 파일 소유권 변경은 리더만 가능

### 4.7 실행 팀원 정리
완료 → shutdown_request.

---

## Phase 5: 검증

### 5.1 자동 검증 (병렬 실행)

변경 영역에 따라 해당 검증을 **병렬 실행**:

| 변경 영역 | 검증 명령 |
|----------|----------|
| TypeScript/JS | `bun run lint` |
| 전체 | `bun run build` |
| 테스트 | `bun run test` |

### 5.2 코드 리뷰

변경 파일 수에 따라 리뷰 방식 결정:

| 변경 파일 수 | 리뷰 방식 |
|-------------|----------|
| 3개 이하 | 리더 직접 리뷰 (Read로 확인) |
| 4개 이상 | `code-review-ai:architect-review` agent 활용 |

### 5.3 검증 게이트 (Phase 6 진입 조건)

5.1 자동 검증 + 5.2 코드 리뷰 결과를 종합 판정한다.
모든 항목이 PASS여야 Phase 6으로 진행. 하나라도 FAIL이면 에러 복구 루프 진입.

| # | 항목 | 통과 기준 | 적용 조건 |
|---|------|----------|----------|
| 1 | 빌드 | `bun run build` 성공 (exit code 0) | 항상 |
| 2 | Lint | `bun run lint` 신규 위반 0건 | 항상 |
| 3 | 테스트 | `bun run test` 전체 통과 | 테스트 변경 시 |
| 4 | 코드 리뷰 | Critical/High 이슈 0건 | 항상 |

### 5.4 에러 복구 루프 (Phase 4 미니루프)

빌드/테스트 실패 시 수정 태스크를 생성하되, **Phase 4의 통제 규칙을 동일하게 적용**한다:
- 파일 소유권 테이블에 수정 대상 파일 등록
- `addBlockedBy`로 의존관계 설정
- 할당된 파일 외 수정 금지 규칙 적용
- 완료 시 handoff-log에 기록

```
실패 감지 → 수정 태스크 생성 (Phase 4 규칙 적용) → 팀원 할당 → 수정 → 재검증
                          ↑                                              ↓
                          └──────────── 3회 미만이면 반복 ───────────────┘
```

- **3회 초과 실패** → 사용자에게 에스컬레이션 (상황 보고 + 롤백 옵션 제시)

### 5.5 검증 결과 처리
- **이슈 없음** → Phase 6 (검증 게이트 통과 확인 후)
- **이슈 발견** → 수정 태스크 생성 (Phase 4 미니루프) → 수정 → 재검증
- **심각한 문제** → 사용자에게 보고, 지시 요청

### 5.6 검증 팀원 정리
완료 → shutdown_request.

---

## Phase 6: 마무리

### 6.1 최종 보고
사용자에게 보고:

```markdown
## 작업 완료 보고

### 요약
{1-2줄 요약}

### 완료 태스크
| # | 태스크 | 상태 |
|---|--------|------|
| 1 | {설명} | ✅ |

### 파일 변경 요약
| 파일 | 변경 유형 | 설명 |
|------|----------|------|
| `path/to/file` | 신규/수정 | {설명} |

### 검증 결과
| 항목 | 결과 |
|------|------|
| 빌드 | ✅ PASS |
| Lint | ✅ PASS |
| 코드 리뷰 | ✅ 이슈 없음 |

### 팀 통계
- 총 팀원: {N}명 (agent_id별 추적)
- 팀원 교체: {N}회
- 복잡도: {S/M/L/XL}
```

### 6.2 커밋 제안
변경 파일 기반으로 커밋 메시지 초안 제시:
- `feat(scope):` / `fix(scope):` / `refactor(scope):` 컨벤션 적용
- scope는 변경 영역에서 유추
- 사용자 승인 후 커밋 진행

### 6.3 팀 정리
남은 팀원 shutdown → TeamDelete.
handoff-log.md의 `last_active`를 최종 업데이트.

## 에러 복구 전략

### 빌드 에러
1. 에러 메시지 분석
2. 수정 태스크 생성 → `general-purpose` (model: `sonnet`) 팀원에게 할당
3. 수정 후 재빌드
4. **3회 초과 실패** → 사용자에게 에스컬레이션

### 테스트 실패
1. 실패 테스트 + 에러 메시지 분석
2. 수정 태스크 생성 → 원래 구현 팀원 또는 새 팀원에게 할당
3. 수정 후 재테스트
4. **3회 초과 실패** → 사용자에게 에스컬레이션

### 에스컬레이션 시 보고 형식

## 팀원 교체 프로세스

**어느 Phase에서든** `HANDOFF:` 수신 또는 `max_turns` 도달 시 즉시:

### 1. Handoff 기록
handoff-log.md에 append:
```markdown
## [{날짜시간}] {팀원} rotation #{n} (Phase: {현재})
### Agent 정보
- agent_id: {agent_id} | agent_type: {agent_type}
### 사유
- {HANDOFF 수신 / max_turns 도달}
### Completed
- {완료}
### In Progress
- {미완료}
### Key Decisions
- {결정사항}
### 파일 변경
- {수정된 파일 목록}
### 테스트 결과
- {마지막 테스트 결과 또는 "미실행"}
```

### 2. 기존 팀원 종료
SendMessage → shutdown_request

### 3. 새 팀원 Spawn
프롬프트 포함 사항:
- 미완료 작업 + 중요 결정
- 태스크 정보 (TaskGet으로 최신 상태 확인)
- 이전 팀원의 파일 변경 목록
- 프로젝트 컨텍스트 (해당 역할 템플릿)
- max_turns 교체 프로토콜
- 파일 충돌 방지 규칙

### 4. 태스크 재할당
TaskUpdate → owner 변경

## max_turns 교체 프로토콜 (모든 팀원 프롬프트에 필수)

```
## max_turns 교체 프로토콜

종료 전에 작업 상태를 보존하세요.

### 태스크 완료 시
1. TaskUpdate로 상태를 completed로 변경
2. 리더에게 완료 메시지 전송

### 태스크 미완료 시 (max_turns 근접 또는 복잡한 작업)
작업이 길어질 것 같으면 중간 안전 지점에서 인수인계하세요:
1. 현재까지 작업한 파일 저장
2. TaskUpdate로 태스크 상태 업데이트
3. 리더에게 메시지:

HANDOFF:
## Completed
- (완료한 것)
## In Progress
- (진행 중인 것, 현재 위치)
## Key Decisions
- (내린 결정)
## 파일 변경
- (수정한 파일 목록)
## Notes
- (후임자가 알아야 할 것)
```

## 리더 컨텍스트 관리

### 리더 자기 보호 규칙
1. **handoff-log.md = 장기 기억** — 중요한 건 모두 기록
2. **팀원 메시지 요약만 기억** — 전문 복사 금지, 핵심 1~3줄
3. **TaskList = ground truth** — 기억 흐리면 TaskList 확인
4. **파일 소유권 = handoff-log 파일 추적 테이블** — 충돌 방지의 근거
5. **Phase 전환마다 handoff-log 업데이트** — 현재 Phase, 완료 상황 기록

### 리더 컨텍스트 소진 대비
리더(현재 세션)도 컨텍스트가 유한하다. 다음을 통해 대비:
- **handoff-log.md에 Phase 전환 시점마다 전체 상태 스냅샷 기록** (현재 Phase, 완료된 태스크, 진행 중 태스크, 남은 태스크)
- **compaction 발생 시**: handoff-log.md를 Read하여 상태 복구 → TaskList와 교차 검증 → 이어서 진행
- **새 세션에서 재개 시**: Phase 1.1 세션 재개 감지 → handoff-log.md + TaskList로 전체 상황 파악 후 중단 지점부터 재개
