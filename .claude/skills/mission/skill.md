# Mission Complete

목표를 받으면 **분석 → 계획 → 실행 → 검증** 사이클을 돌며, 모든 검증 게이트를 통과할 때까지 멈추지 않는다.
팀 스킬과 달리 단일 에이전트가 처음부터 끝까지 책임진다.

인자로 전달된 내용이 미션 설명이다. 즉시 Phase 1부터 시작하라.

---

## 전체 흐름

```
Phase 1: 분석     목표 파악, 코드베이스 탐색, 영향 범위 파악, baseline 캡처
   ↓
Phase 2: 계획     구체적 실행 계획 수립 → 사용자 승인
   ↓
Phase 3: 실행     계획대로 구현
   ↓
Phase 4: 검증     자동 검증 + 코드 리뷰 → 실패 시 수정 루프
   ↓
Phase 5: 완료     최종 보고
```

**핵심 원칙: Phase 4 검증을 100% 통과하기 전에는 절대 완료로 넘어가지 않는다.**

---

## /team 전환 규칙

Phase 1 분석 후, 다음 조건 중 하나라도 해당하면 `/team` 전환을 사용자에게 제안:
- 영향 파일 10개 이상
- Frontend + Backend + DB 마이그레이션 동시 변경
- 서로 독립적인 작업이 3개 이상 (병렬 처리 이득)

사용자가 `/mission`으로 계속하겠다고 하면 그대로 진행.

## Phase 1: 분석

### 1.1 목표 명확화
미션 설명에서 다음을 추출:
- **What**: 무엇을 달성해야 하는가
- **Why**: 왜 필요한가 (맥락)
- **Scope**: 어디까지가 범위인가

애매한 부분이 있으면 AskUserQuestion으로 확인. **추측하지 말고 물어봐라.**

### 1.2 코드베이스 탐색
Explore agent 또는 직접 Glob/Grep/Read로:
- 관련 파일 전수 조사
- 기존 패턴 파악
- 의존 관계 매핑

### 1.3 영향 범위 정리
```
| 파일 | 변경 유형 | 이유 |
|------|----------|------|
| `path/to/file` | 수정/신규 | {이유} |
```

### 1.4 Baseline 캡처
변경 영역에 해당하는 검증 명령을 **구현 전에** 실행하여 기존 상태를 기록:

```
## Baseline (구현 전)
| 검증 | 결과 | 기존 실패 |
|------|------|----------|
| eslint | ✅ 0건 / ❌ N건 | {기존 위반 목록} |
| vitest | ✅ 전체 통과 / ❌ N건 실패 | {기존 실패 테스트} |
| next build | ✅ / ❌ | {기존 에러} |
```

**이 baseline이 Phase 4 검증의 기준선이 된다.**
- 기존에 깨진 테스트 = 이번 미션의 책임 아님 (신규 회귀만 잡는다)
- baseline 자체가 red면 사용자에게 알리고 "신규 회귀 0건"을 통과 기준으로 조정

## Phase 2: 계획

### 2.1 실행 계획 수립
분석 결과 기반으로 작성:

```markdown
## 미션 계획

### 목표
{1-2줄 요약}

### 영향 파일
| # | 파일 | 변경 내용 |
|---|------|----------|
| 1 | `path` | {설명} |

### 구현 단계
1. {단계1}: {설명}
2. {단계2}: {설명}
3. ...

### 검증 기준
- [ ] {검증1}: {구체적 통과 기준}
- [ ] {검증2}: {구체적 통과 기준}

### 리스크
- {잠재 문제}
```

**검증 기준은 기계적으로 확인 가능해야 한다:**
- Bad: "코드가 깔끔해야 한다" → 주관적
- Good: "`bun run build` exit code 0" → 자동 확인 가능
- Good: "`bun run test` 전체 통과" → 자동 확인 가능
- Good: "관리자 페이지에서 {기능}이 정상 작동" → 수동이지만 구체적

### 2.2 바로 실행
계획을 간략히 출력하고 **승인 없이 즉시 Phase 3으로 진행한다.**
- 계획이 명확하면 → 바로 실행
- 선택지가 여러 개이고 어느 쪽이 맞는지 모호할 때만 → 사용자에게 질문

---

## Phase 3: 실행

### 3.1 단계별 구현
계획의 구현 단계를 순서대로 실행.
각 단계 완료 시 간단히 진행 상황 보고:

```
[1/5] ✅ {단계1} 완료
[2/5] 🔄 {단계2} 진행 중...
```

### 3.2 실행 중 원칙
- 계획에 없는 변경은 하지 않는다 (단, Phase 4 수정 루프에서의 버그 수정은 예외)
- 예상 못한 문제 발견 시 → 계획 수정이 필요하면 사용자에게 보고
- 기존 코드 스타일/패턴을 따른다 (CLAUDE.md 규칙 준수)

---

## Phase 4: 검증

### 4.1 자동 검증 (변경 영역에 따라 해당 항목 실행)

변경된 파일 경로를 기준으로 검증 대상을 자동 판별한다:

| 변경 영역 | 검증 명령 | 통과 기준 |
|----------|----------|----------|
| TypeScript/JS 전체 | `bun run lint` | baseline 대비 신규 위반 0건 |
| 테스트 관련 | `bun run test` | baseline 대비 신규 실패 0건 |
| 전체 | `bun run build` | exit code 0 |
| UI (`src/app/`, `src/components/`) | 브라우저 스모크 테스트 (아래 4.1.1 참조) | 주요 플로우 정상 렌더링 + 인터랙션 성공 |

**병렬 실행 가능한 검증은 병렬로 돌린다.**

#### 4.1.1 브라우저 스모크 테스트 (Frontend UI 변경 시)

프론트엔드 UI 파일(`src/app/`, `src/components/`)이 변경된 경우에만 실행한다.

**판별 기준:**
- 단순 UI 변경 (버튼, 텍스트, 스타일 등) → `agent-browser` 스킬로 해당 페이지만 스모크 테스트
- 복잡한 플로우 변경 (결제, 폼 제출, 멀티스텝 등) → `dogfood` 스킬로 해당 플로우 탐색 테스트
- 백엔드만 변경, 또는 설정/유틸만 변경 → **스킵**

**실행 방법:**
1. 로컬 dev 서버가 실행 중인지 확인 (`localhost:3000`)
2. 변경된 페이지/컴포넌트에 해당하는 URL을 특정
3. `agent-browser` 또는 `dogfood` 스킬을 호출하여 테스트
4. 스크린샷 기반으로 렌더링 정상 여부 판단

**통과 기준:**
- 페이지가 에러 없이 렌더링됨
- 변경한 UI 요소가 화면에 표시됨
- 주요 인터랙션(클릭, 입력 등)이 정상 동작함

### 4.2 코드 자체 검증
변경한 파일을 다시 Read하여 다음을 확인:
- 오타, 누락된 import, 미사용 변수
- CLAUDE.md 규칙 위반 여부
- 보안 취약점 (OWASP Top 10)
- 기존 패턴과의 일관성

### 4.3 사용자 정의 검증
Phase 2에서 수립한 **검증 기준** 항목을 하나씩 체크:
- 자동 확인 가능한 항목 → 실행하여 확인
- 수동 확인 필요한 항목 → 사용자에게 확인 요청

### 4.4 검증 게이트

**모든 항목이 PASS여야 Phase 5로 진행:**

```
## 검증 결과
| # | 항목 | 결과 | 비고 |
|---|------|------|------|
| 1 | Lint (eslint) | ✅ PASS | 신규 위반 0건 |
| 2 | 테스트 (vitest) | ✅ PASS | 신규 실패 0건 |
| 3 | 빌드 (next build) | ✅ PASS | |
| 4 | 코드 리뷰 | ✅ PASS | |
| 5 | {사용자 정의 검증1} | ✅ PASS | |
```

### 4.5 수정 루프 (FAIL 시)

```
검증 실패 → 원인 분석 → 수정 → 재검증
      ↑                          ↓
      └──── 5회 미만이면 반복 ────┘
```

1. 실패 원인을 정확히 분석
2. 최소 범위로 수정 (다른 것 건드리지 않음)
3. 수정 후 **실패한 검증을 먼저 재실행** (빠른 피드백)
4. 해당 검증 통과 시 → **전체 검증을 다시 실행** (회귀 확인)
5. **동일 실패 시그니처 2회 연속** → 접근 방식을 바꿔라. 같은 수정을 반복하지 마라.
6. **5회 초과 실패** → 사용자에게 에스컬레이션:

```
⚠️ 자동 수정 실패 (시도 {N}회)

실패 내용:
- {에러 요약}

시도한 수정:
1. {수정1} → {결과}
2. {수정2} → {결과}

옵션:
1. 수동 개입 후 재검증
2. 해당 변경 제외하고 나머지만 완료
3. 전체 롤백
```

## Phase 5: 완료

### 5.1 최종 보고

```markdown
## 미션 완료

### 요약
{1-2줄}

### 변경 파일
| 파일 | 변경 | 설명 |
|------|------|------|
| `path` | 수정/신규 | {설명} |

### 검증 결과
| 항목 | 결과 |
|------|------|
| Lint | ✅ |
| 테스트 | ✅ |
| 빌드 | ✅ |
| {사용자 정의} | ✅ |

### 수정 루프
- 검증 시도: {N}회
- 수정 횟수: {N}회
```

### 5.2 커밋 제안
변경 내용 기반으로 커밋 메시지 초안 제시.
사용자 승인 시에만 커밋. (CLAUDE.md 규칙: 명시적 요청 없이 커밋 금지)

---

## 에스컬레이션 규칙

다음 상황에서는 **즉시 사용자에게 보고**하고 지시를 기다린다:
- 계획에 없는 대규모 변경이 필요할 때
- 기존 기능이 깨질 위험이 있을 때
- DB 스키마 변경, 결제/인증 로직 수정이 필요할 때
- 검증 루프 5회 초과 실패
- 동일 실패 시그니처 2회 연속 (접근 방식 변경 필요)
- 요구사항 자체에 모순이 있을 때

**절대 혼자 판단해서 넘어가지 마라. 확인하고 넘어가라.**
